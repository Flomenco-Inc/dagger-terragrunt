# AGENTS.md — dagger-terragrunt

See [flo-docs/AGENTS.md](https://github.com/Flomenco-Inc/flo-docs/blob/main/AGENTS.md)
for shared conventions. This file is the repo-local supplement.

## Module overview

Thin Dagger wrapper around `terragrunt run-all plan/apply` with OIDC-based
AWS role assumption performed **inside the container**. All logic in
`main.go`.

Sister module: [`dagger-ci`](https://github.com/Flomenco-Inc/dagger-ci). The
two are intentionally separate because their invariants differ — `dagger-ci`
is hermetic + credential-free; `dagger-terragrunt` is credential-mandatory.
Conflating them would invite credential leakage into the CI module by
accident.

## File organization

- `main.go` — module entry + all functions + helpers.
- `dagger.json` — engine pin (v0.20.6) + SDK selection (Go).
- `go.mod` — minimal; regenerated by `dagger develop`.
- `internal/dagger/` — generated, not committed.
- `dagger.gen.go` — generated, not committed.

## Credential contract

The module accepts **exactly two** credential-shaped inputs at its public
API boundary:

1. `--role-arn` — plain string. The IAM role ARN in the target AWS account.
2. `--oidc-token` — `*dagger.Secret`. A GitHub Actions-minted JWT for the
   `sts.amazonaws.com` audience.

Inside the container (`runTerragrunt` in `main.go`) the token is mounted
at `/run/secrets/oidc-token` and exchanged via
`aws sts assume-role-with-web-identity` for temporary session credentials.
Those credentials are exported as env vars **for the single terragrunt
exec only** and never logged, never cached, never returned to the caller.

This design is non-negotiable:

- No function accepts `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` directly.
- No function accepts a generic "creds" secret that could conceal long-lived
  IAM-user keys. The `oidc-token` name is prescriptive on purpose.
- No environment-variable fallback. If the OIDC token is missing the call
  fails fast with a clear error — it does not silently fall back to
  ambient `AWS_*` env vars on the runner.

If you find yourself adding an AWS-touching function, reuse the existing
`runTerragrunt` helper. Do not write a new credential plumbing path.

## Why we do the OIDC exchange inside the container, not on the runner

Historically the flo terragrunt workflows used
`aws-actions/configure-aws-credentials` to exchange the OIDC token on the
GitHub runner, then passed the resulting `AWS_*` env vars into Dagger. That
worked, but it had three problems:

1. The session credentials lived in the runner environment for the whole
   job, readable by any subsequent step (including a compromised third-
   party action).
2. The Dagger function signature had to accept raw access keys, making it
   syntactically possible for a caller to pass a static IAM user key.
3. The GHA runner-side credential lifetime could not be capped below the
   IAM role's `MaxSessionDuration`; callers had to remember to set a
   short `role-duration-seconds`.

Moving the exchange inside the Dagger container eliminates all three. The
runner only ever holds the OIDC JWT (tied to `sts.amazonaws.com` audience,
useless for anything other than assuming the specified role); the session
credentials exist only for the duration of one `WithExec` call.

## What NOT to do

- **Do not add a function that accepts `AWS_ACCESS_KEY_ID` / similar.**
  Even as a "just for testing" variant. The module's public API is the
  policy surface; if it can be used with static keys, it will be.
- **Do not auto-detect the env.** The `env` parameter is required on every
  apply-capable function. An auto-detection heuristic (e.g. "if branch is
  `main` apply prd") is exactly the footgun that once-in-a-career apply-to-
  wrong-env mistakes are made of. Keep it explicit.
- **Do not add an `--auto-approve=false` path to `apply`.** The function is
  deliberately "apply with auto-approve". The approval gate lives in the
  GitHub Actions environment rule; by the time this function is invoked,
  approval has already happened. An interactive apply would deadlock in CI
  (stdin is closed) and signals that someone is about to run this from a
  laptop against prod — which should never happen and is better prevented
  by the IAM role trust policy allowing only the GHA OIDC SAR.
- **Do not add `destroy` to v0.x.** Destroy is high-risk and should be a
  multi-step ceremony (manual plan → manual confirm → manual apply via
  terragrunt directly, not via CI). Add it only if a clear recurring
  need arises, and then with a mandatory `--confirm=<env>` arg that must
  match the `env` param.
- **Do not couple this module to flo-specific env names.** `env` is free-
  form; the module just cds into `./<env>/`.
- **Do not commit `internal/dagger/` or `dagger.gen.go`.**
- **Do not print `$creds` or any intermediate shell variable** inside the
  runTerragrunt script. The script deliberately uses `set -eu` (not `-x`)
  to avoid tracing the sts exchange line.
- **Do not bump `--duration-seconds` default above 900.** The IAM role's
  `MaxSessionDuration` should cap it at 900 anyway, but the default is
  belt-and-suspenders. Longer-lived session creds = larger blast radius
  if the JWT or intermediate container layer leaks.

## Local testing

`validate` runs with no credentials:

```bash
dagger develop
dagger functions

dagger call validate --src=../flo-account-admin
```

`plan` and `apply` require an OIDC JWT that only GitHub Actions can mint.
There is **no supported local path** for them. If you need to iterate on
the plan/apply code path while developing this module:

1. Cut a draft PR on a consuming repo (e.g. flo-account-admin).
2. Push trivial commits that re-trigger the plan workflow.
3. Watch CI logs for the Dagger invocation.

Yes, this is slower than a local loop. That's the point — it makes an
accidental "dagger call apply from my laptop against prod" physically
impossible.

## Versioning

Same discipline as `dagger-ci`:

- Patch: tool-version bump (terraform, terragrunt, AWS CLI via rebuild).
- Minor: new function or backwards-compatible arg addition.
- Major: breaking signature change.

Tag releases with `git tag v0.1.0 && git push --tags`. The OIDC-native
design is the initial public release; no pre-v0.1.0 history exists.
